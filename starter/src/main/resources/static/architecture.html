硬件部署架构

面向企业版与平台版集群部署，推荐将接入层、应用层、消息中间件、数据存储与文件存储拆分到独立节点或独立资源池中。多个微语服务实例共享同一套消息总线与存储资源，通过负载均衡统一接入，对外提供高可用、可横向扩展的客服与 IM 服务。

部署原则

开源社区版适合单机 企业版建议 3 个以上应用节点 所有节点共享同一消息中间件 文件存储配置必须统一
蓝色实线：同步直连 橙色虚线：消息广播与状态同步 灰色虚线：异步落库与处理

集群部署拓扑图

通过一张主图展示终端入口、负载均衡、应用集群、消息中间件、数据存储与文件存储之间的同步链路、广播链路和异步链路。

SVG 风格拓扑
链路说明
同步直连：鉴权、查询、配置、缓存、文件访问 消息广播：实时聊天、客服上下线、状态同步、系统通知 异步处理：消息消费、任务处理、异步落库、削峰填谷
落地建议
消息中间件建议内网地址 所有应用节点同构配置 文件存储必须共享同一配置 生产环境配套监控与备份

推荐部署规格：3 节点最小生产拓扑

适合中小规模正式环境，目标是以最少机器满足高可用基础能力。建议将入口、应用和数据角色做最小拆分，避免全部混布在单机上。

节点 1：Nginx / WAF / 反向代理 节点 2：应用实例 A + 应用实例 B 节点 3：MySQL + Redis + Artemis + MinIO
适合初期生产 建议额外做数据库备份 应用至少双实例

推荐部署规格：5 节点标准生产拓扑

适合稳定运营和持续扩容场景。入口、应用、中间件、数据、文件存储各自独立，故障域更清晰，后续横向扩展成本更低。

节点 1：Nginx / 云负载均衡接入 节点 2：应用实例 A 节点 3：应用实例 B 节点 4：Artemis / RabbitMQ 节点 5：MySQL + Redis + ElasticSearch + MinIO
更适合正式商用 支持继续扩应用节点 中间件与业务隔离
